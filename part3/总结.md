# Part 3 Raft 测试日志分析总结

## 一、总体说明

Part 3 的测试主要验证：**在网络故障或节点宕机的情况下，Raft 如何保证共识正确性与日志一致性**。

核心场景包括：
- **网络分区（Disconnect）**：节点仍运行，但无法与其他节点通信
- **节点崩溃（Crash）**：节点完全停止，依赖持久化 Storage 恢复
- **日志冲突**：分区导致不同节点产生分歧日志，需通过 ConflictIndex/ConflictTerm 覆盖

Part 3 在 Part 2 基础上增加了**持久化（currentTerm、votedFor、log）**，使节点在崩溃重启后能从 Storage 恢复状态，继续参与共识。

---

## 二、测试用例与验证要点

### 1. TestCrashThenRestartFollower

| 项目 | 说明 |
|------|------|
| **场景** | 崩溃一个 Follower，随后重启 |
| **流程** | 集群提交 5、6、7 → Crash Follower 2 → 0、1 继续运行 → RestartPeer(2) 用 storage[2] 创建新 Server |
| **验证点** | 2 从 Storage 恢复 log，收到 Leader 的 LeaderCommit 后更新 commitIndex，正确提交 5、6、7 |
| **要点** | 持久化使 Follower 重启后无需补全 log，只需同步 commitIndex |

---

### 2. TestCrashThenRestartLeader

| 项目 | 说明 |
|------|------|
| **场景** | 崩溃 Leader，随后重启 |
| **流程** | 1 为 Leader，提交 5、6、7 → Crash 1 → 0、2 选举，0 成为新 Leader → RestartPeer(1) |
| **验证点** | 1 恢复 term=1、log，收到 0 的 AppendEntries（term=2）后更新 term 并转为 Follower，通过 LeaderCommit 提交 5、6、7 |
| **要点** | 旧 Leader 重启后必须接受更高 term，降级为 Follower；log 已持久化，无需补日志 |

---

### 3. TestCrashThenRestartAll

| 项目 | 说明 |
|------|------|
| **场景** | 全集群崩溃，随后依次重启 |
| **流程** | 提交 5、6、7 → Crash 1、2、0 → Restart 1、2、0 → 1 先超时成为 Leader（term=2）→ 提交 8 |
| **验证点** | 所有节点从 Storage 恢复，重新选举，新 Leader 能继续提交新命令 |
| **要点** | commitIndex 不持久化，重启后由 Leader 根据 matchIndex 重新确定；lastApplied=-1 时从 index 0 开始应用 |

---

### 4. TestReplaceMultipleLogEntries

| 项目 | 说明 |
|------|------|
| **场景** | Leader 被隔离后产生未复制日志，新 Leader 选举后覆盖冲突日志 |
| **流程** | 1 为 Leader，提交 5、6 → Disconnect 1 → 1 收 21–24（未复制）→ 0 成为 Leader，提交 8、9、10 → Crash 0 → Restart 0 → 2 成为 Leader → 2 向 1 发 AppendEntries，1 返回 ConflictIndex/ConflictTerm → 2 退回到 PrevLogIndex=1，成功覆盖 21–24 |
| **验证点** | ConflictIndex/ConflictTerm 机制正确回溯，1 的冲突日志被 8、9、10 覆盖 |
| **要点** | PrevLogIndex = nextIndex-1，PrevLogTerm = log[PrevLogIndex].Term；Leader 根据 ConflictTerm 回溯 nextIndex |

---

### 5. TestCrashAfterSubmit

| 项目 | 说明 |
|------|------|
| **场景** | Leader 在 Submit 后、收到多数派确认前崩溃 |
| **流程** | 0 为 Leader，Submit 5，发出 AppendEntries 后立即 Crash → 1、2 已收到并写入 5 → 1、2 选举，2 成为 Leader → Restart 0 → 2 提交 6，同时推进 5、6 的 commitIndex |
| **验证点** | 5 虽未在 0 上提交，但已复制到 1、2；新 Leader 2 的 log 含 5，能正确提交 5、6 |
| **要点** | 只要条目已复制到多数节点，Leader 崩溃不影响最终一致性 |

---

### 6. TestDisconnectAfterSubmit

| 项目 | 说明 |
|------|------|
| **场景** | Leader 在 Submit 后、收到多数派确认前被网络隔离 |
| **流程** | 2 为 Leader，Submit 5，发出 AppendEntries 后 Disconnect 2 → 0、1 已收到并写入 5 → 0、1 选举，1 成为 Leader → Reconnect 2 → 2 收到 1 的心跳，转为 Follower → 1 提交 6，5、6 在 0、1、2 上提交 |
| **验证点** | 与 TestCrashAfterSubmit 类似，但 2 仍运行；重连后 2 接受更高 term，log 已含 5 |
| **要点** | Disconnect 仅断网，Crash 为完全停止；两者在「条目已复制到多数」时行为一致 |

## 三、核心机制小结

### 1. 持久化

| 持久化 | 不持久化 |
|--------|----------|
| currentTerm | commitIndex |
| votedFor | lastApplied |
| log[] | nextIndex、matchIndex |

commitIndex 由 Leader 根据 matchIndex 计算，Follower 通过 LeaderCommit 接收，重启后可安全恢复。

### 2. 日志冲突与覆盖

- **PrevLogIndex / PrevLogTerm**：校验 Follower 与 Leader 的前缀一致性
- **ConflictIndex / ConflictTerm**：Follower 返回冲突信息，Leader 回溯 nextIndex
- **覆盖规则**：Leader 的 log 代表多数派共识，Follower 的冲突部分被覆盖

### 3. 网络故障与宕机

| 类型 | 节点状态 | 恢复方式 |
|------|----------|----------|
| Disconnect | 仍运行，无法通信 | Reconnect |
| Crash | 完全停止 | RestartPeer + Storage 恢复 |

### 4. 角色转换

- **旧 Leader 重启/重连**：收到更高 term 的 AppendEntries 后，更新 term 并转为 Follower
- **Follower 重启**：从 Storage 恢复 log，通过 LeaderCommit 更新 commitIndex

## 四、测试覆盖矩阵

| 故障类型 | 故障节点 | 恢复方式 | 对应测试 |
|----------|----------|----------|----------|
| Crash | Follower | RestartPeer | TestCrashThenRestartFollower |
| Crash | Leader | RestartPeer | TestCrashThenRestartLeader |
| Crash | 全部 | RestartPeer | TestCrashThenRestartAll |
| Disconnect | Leader | Reconnect | TestReplaceMultipleLogEntries（部分）、TestDisconnectAfterSubmit |
| Crash | Leader | RestartPeer | TestCrashAfterSubmit |
| Disconnect | Leader | Reconnect | TestDisconnectAfterSubmit |

---

## 五、结论

Part 3 的测试表明：**在网络分区和节点宕机下，Raft 通过持久化、日志冲突解决和正确的角色转换，能够保持共识正确性与日志一致性**。只要条目已复制到多数节点，即使 Leader 崩溃或断网，新 Leader 也能继续推进并完成提交。
